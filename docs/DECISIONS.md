# Blanoir 설계 결정 (DECISIONS)

> 무엇을 왜 만들기로 했나. 코드 작성 규칙은 `AGENT.md`, 현황/작업순서는 `../CLAUDE.md` 참고.

## 프로젝트 개요

비개발자도 5분 안에 자기 홈페이지를 만들고 배포할 수 있는 **한글 노코드 페이지 빌더**.

## 서비스 정체성

### 서비스명: **Blanoir** (블라누아)

**의미:**

- 프랑스어 `Blanc`(흰색) + `Noir`(검정)의 합성어
- "흰 캔버스 위에 검정 잉크로 표현하다"
- "빈 페이지에 당신만의 이야기를 새기다"
- 모노톤 디자인 철학 직접 담음

**한 줄 소개:**

> "5분 만에 만드는 내 홈페이지. 코딩 모르는 사람도, 한글 클릭만으로, 무료로."

### 타겟 사용자

타겟 안 좁힘 = "코딩 모르는데 자기 페이지 만들고 싶은 모든 사람"

- 자기소개/포트폴리오
- 카페/매장 소개
- 결혼식 청첩장
- 학생 이력서
- 취미/개인 페이지

### 차별점

| 비교 대상     | 우리 차별점                |
| ------------- | -------------------------- |
| 아임웹/식스샵 | 가볍고 무료, 비개발자 친화 |
| 노션 사이트   | 디자인 자유로움, 한글 특화 |
| Wix/Webflow   | 가벼움, 한글, 무료         |

핵심 가치: **"학습 없이 직관적으로"** + **무료** + **한글** + **빠른 배포**

## 기술 선택 이유

> 스택 목록은 `../CLAUDE.md` 참고. 여기는 왜 그렇게 골랐는지.

### 풀스택 선택 이유

- 우리 서비스 백엔드 가벼움 (CRUD + 인증)
- 개발 속도, 단순함, 비용 0원
- SSR로 페이지 조회 처리 자연스러움 (DB 직접 호출)

### 서버리스 커넥션 관리 (중요)

Vercel은 서버리스 — 함수가 요청마다 켜졌다 꺼짐. DB 커넥션 캐싱이 필수.

- 캐싱 없으면: 호출마다 `mongoose.connect()` → 커넥션 누수 → MongoDB Atlas 무료(M0) 한도(약 500개) 초과 시 연결 실패
- 해결: `lib/mongoDB.ts`에서 `global` 객체에 커넥션 캐싱 후 재사용 (Next.js + Mongoose 공식 표준 패턴)
- 옵션 아니라 정석 — 트래픽 적어도 무조건 적용
- Blanoir는 단순 CRUD라 무거운 집계·크롤러가 없어 커넥션 폭증 위험 자체가 낮음. 캐싱 패턴만 지키면 충분

## 데이터 모델 (MongoDB)

### User 스키마

```ts
{
  _id: ObjectId,
  email: String,           // 로그인 이메일 (unique)
  password: String,        // 로컬 가입시만 (bcrypt 해시). 소셜 가입은 null
  resetCode: String,       // 비번 재설정 인증코드 (bcrypt 해시). 미발급시 null
  resetCodeExpiresAt: Date,// 인증코드 만료 시각
  resetCodeAttempts: Number,// 인증코드 검증 실패 횟수 (5회 초과 시 코드 폐기)
  name: String,            // 닉네임 (한글 OK, 화면 표시용)
  handle: String,          // URL용 (영문+숫자+언더바, 자동 생성)
  profileImage: String,    // 프로필 이미지 URL (소셜은 자동, 로컬은 기본 이미지)
  provider: String,        // "google" | "kakao" | "naver" | "local"
  createdAt: Date,
  updatedAt: Date
}
```

### Page 스키마

```ts
{
  _id: ObjectId,
  pageId: String,          // nanoid 10자 (URL용)
  userId: ObjectId,        // 작성자
  title: String,           // 페이지 제목 (한글 OK, 자유롭게)
  sections: Section[],     // 섹션 데이터 배열
  isPublic: Boolean,       // 공개/비공개 (기본 false)
  createdAt: Date,
  updatedAt: Date
}
```

### Section 데이터 (예시)

```ts
{
  id: "abc123",            // 섹션 고유 ID
  type: "title" | "paragraph" | "image" | "button" | "divider" | "spacer" | "gallery" | "card",
  content: { ... },        // 타입별 콘텐츠
  style: { ... },          // 타입별 스타일
  container?: {            // 모든 섹션 공통 박스 (없으면 미저장)
    height?: number,       // 콘텐츠보다 크게 늘린 박스 높이(px)
    backgroundColor?: string, // 배경색·그레디언트 (빈값=투명)
    backgroundImage?: string, // 배경 이미지 URL
    animation?: "none" | "up" | "down" | "left" | "right" | "fade" | "zoom" | "blur" // 스크롤 등장 효과 (기본 none)
  }
}
```

#### 섹션별 예시

**제목 (title):**

```ts
{
  id: "s1",
  type: "title",
  content: { text: "안녕하세요" },
  style: {
    size: "small" | "medium" | "large" | "xlarge",
    color: "#000000",
    align: "left" | "center" | "right",
    bold: true,
    italic: false,
    font: "pretendard"
  }
}
```

**문단 (paragraph):** title과 유사, text 위주

**이미지 (image):** `alt`은 업로드 파일명에서 자동 생성(수동 입력 X — 비개발자 친화)

```ts
{
  type: "image",
  content: { src: "https://cloudinary.../...", alt: "..." },
  style: {
    size: "small" | "medium" | "large",   // 박스 폭, 콘텐츠 가로폭 대비 비율 1/3 / 2/3 / 전체
    shape: "square" | "rounded" | "circle",
    align: "left" | "center" | "right",    // 박스가 좁을 때 가로 위치
    ratio: "original" | "square" | "wide", // 원본=자연비율, 그 외=프레임에 cover로 꽉 채움
    zoom: 1,                               // 1~3배 확대
    focusX: 50, focusY: 50                 // 보일 부분(초점) %, 캔버스에서 드래그로 조정
  }
}
```

**버튼 (button):** 라벨·URL 모두 스타일 패널에서 입력(인라인 편집 X — 컨트롤 성격)

```ts
{
  type: "button",
  content: { text: "버튼", url: "https://..." },
  style: {
    color: "",                         // 배경색, 빈값=기본(--foreground)
    textColor: "",                     // 글자색, 빈값=기본(--background)
    shape: "square" | "rounded",
    size: "small" | "medium" | "large",
    width: "auto" | "wide" | "full",   // wide=50%, full=100%
    align: "left" | "center" | "right"
  }
}
```

**구분선 (divider):** content 없음, style만

```ts
{
  type: "divider",
  content: {},
  style: {
    variant: "solid" | "dashed" | "dotted",
    thickness: "thin" | "medium" | "thick",
    color: ""                          // 빈값=기본 보더색(--border)
  }
}
```

**공백 (spacer):** 섹션 사이 여백 높이만 주는 섹션. content 없음, style만(divider 패턴). 평소 투명, 에디터에선 hover/선택 시에만 영역이 보여 클릭·삭제·순서변경 가능.

```ts
{
  type: "spacer",
  content: {},
  style: {
    size: "small" | "medium" | "large"   // 높이 24 / 48 / 96px
  }
}
```

**갤러리 (gallery):** 여러 이미지를 **한 줄 가로 캐러셀**로 표시. 칸 수 컨트롤은 두지 않고 한 줄 최대 3개를 채운다(이미지가 적으면 남는 칸 없이 폭을 채움, 빈 공간 X. 컨테이너 폭이 좁으면 `ResizeObserver`로 2개). 3개보다 많으면 호버 시 좌우 화살표로 가로 슬라이드(끝단에선 해당 방향 화살표 숨김, 스크롤바 숨김 + 스냅). 추가·교체·삭제·순서변경은 모두 스타일 패널에서. 빈 상태는 캔버스 스켈레톤 그리드(첫 업로드). 각 이미지는 `alt` 포함 객체.

> 칸 수를 사용자가 고르지 않는 이유: "폭을 꽉 채움"과 "크기 버튼이 항상 변함"은 적은 이미지에선 양립 불가(한 줄에 N개를 빈틈없이 채우면 각 폭이 N으로 고정됨). 비개발자 친화상 "눌러도 안 변하는" 혼란을 없애려고 컨트롤 자체를 제거하고 개수 기반 자동 배치로 통일. 2-up/4-up 등 레이아웃 다양성은 아직 안 만듦(갤러리 캐러셀 모드).

```ts
{
  type: "gallery",
  content: { images: [{ url: "...", alt: "..." }, ...] },
  style: {
    displayMode: "grid",
    shape: "square" | "rounded" | "circle",
    gap: "none" | "small" | "medium" | "large"
  }
}
```

**카드 (card):** 한 섹션에 **여러 카드**. 카드 추가·삭제·순서변경·이미지/제목/설명 편집은 모두 스타일 패널에서(컨트롤 성격, 인라인 X). 레이아웃은 세로형(이미지 위·텍스트 아래)/가로형(이미지 왼·텍스트 오른)/그리드형(한 줄 여러 개). 갤러리와 같은 원칙으로 열 수 컨트롤은 두지 않고, 그리드는 한 줄 최대 3개를 카드 수에 맞춰 채우며 넘으면 **다음 줄로 줄바꿈**(컨테이너 폭을 `ResizeObserver`로 재 좁으면 2개, 모바일 미리보기 박스도 반영), 같은 행 카드끼리 제목·설명 영역 높이를 가장 긴 것에 맞춰 통일. 빈 상태는 캔버스 스켈레톤 그리드(첫 업로드)로 카드 생성. `alt`은 파일명 자동.

```ts
{
  type: "card",
  content: { cards: [{ id, image, alt, title, description }, ...] },
  style: {
    layout: "vertical" | "horizontal" | "grid",
    align: "left" | "center" | "right"
  }
}
```

| ------------- | ------------- | ------------------- |
| 김철수        | kimchulsoo    | /user/kimchulsoo    |

- **nanoid 10자** (예: `V1StGXR8_Z`)
- 추측 불가 (보안)
- 충돌 확률 거의 0
- 노션과 유사한 패턴

## 회원가입 / 로그인 흐름

### 소셜 로그인 (구글/카카오/네이버) — 원클릭 가입

1. "구글로 로그인" (또는 카카오/네이버) 클릭
2. 소셜 인증 통과
3. 자동 가입:
   - name = 소셜 이름 그대로
   - handle = name 기반 자동 변환 + 중복시 숫자
   - profileImage = 소셜 사진 그대로
   - email = 소셜 이메일
   - provider = "google" | "kakao" | "naver"
4. 바로 대시보드로 이동

### 로컬 회원가입 (이메일/비번) — 폼 입력

1. 회원가입 페이지 진입
2. 입력: 이메일 + 비번 + 닉네임 (3개)
3. 검증:
   - 이메일 형식 + 중복 체크
   - 비번 8자 이상
   - 닉네임 한글 OK
4. 비번 bcrypt 해싱 → DB 저장
5. handle 자동 생성
6. provider = "local"
7. 자동 로그인 → 대시보드

### 로컬 로그인

- NextAuth `CredentialsProvider` 사용
- 이메일 + 비번 → bcrypt 검증 → JWT 발급

### 비밀번호 재설정 (MVP 포함) — 이메일 인증코드 방식

1. 사용자가 이메일 입력
2. 이메일이 DB에 있으면(local 계정) → **6자리 숫자 인증코드 생성**
3. 인증코드 bcrypt 해싱 → DB의 `resetCode` 저장 + `resetCodeExpiresAt`(10분) 설정
4. 인증코드를 이메일로 발송 (Gmail SMTP / nodemailer, 키 없으면 콘솔 폴백)
5. 사용자가 인증코드 입력 → 검증 (일치 + 미만료)
6. 통과 시 새 비밀번호 입력 → bcrypt 해싱 → `password` 업데이트 + `resetCode`/`resetCodeExpiresAt` 폐기

> 미가입/소셜 계정이면 토스트로 안내하고 발송 중단 (데모 — UX 우선, 계정 열거 노출은 감수. 운영 시 항상 성공 응답으로 가릴 수 있음)
> 이메일 = ID이므로 **아이디 찾기는 없음**
> 인증코드 **5회 오입력 시 폐기**(brute-force 방지) — `resetCodeAttempts`로 카운트
> 이메일 인증(가입 시)은 아직 안 만듦. 닉네임/handle 변경은 설정 페이지(`/settings`)에서 가능

## 에디터 UX

### 입력 방식: 카드/섹션 추가 (아임웹 스타일이지만 더 가볍게)

- `[+ 새 섹션 추가]` 버튼 → 종류 메뉴 (제목/문단/이미지/버튼/구분선/갤러리/카드)
- 섹션 추가시 캔버스에 표시
- 학습 0, 클릭만으로 작동

### 시작 템플릿

빈 캔버스 공포를 없애려 에디터 왼쪽에 종류별 시작 템플릿을 둔다.

- **5종**: 프로필 / 매장 / 청첩장 / 포트폴리오 / 이력서. 타겟 사용 사례를 빠짐없이 덮으면서, 더 많으면 고르는 것부터 부담이라 5개로 제한. 라벨은 **서로 단어가 겹치지 않게** 짓고(자기소개·이력서·포트폴리오의 의미 중복을 형태로 분리: 프로필=링크 모음, 포트폴리오=카드 전시, 이력서=글 정리) 각자 목적·형태가 한눈에 갈리게 한다.
- 템플릿 = **골격 + 안내 문구 + 빈 이미지 자리**. 글·제목·버튼·구분선은 placeholder성 실제 문구까지 채우고, 이미지/갤러리/카드는 빈 상태(첫 업로드는 캔버스). 이미지는 사용자별 Cloudinary 폴더라 템플릿에 못 박아 둔다.
- **배경·등장효과는 안 넣는다**: 취향이 강한 선택이라 미리 깔면 5개가 비슷해 보이고 "내 것 같지 않다". 색·배경은 사용자가 배경 탭에서 입히는 분업. 템플릿은 중립 골격까지만.
- **공통 푸터도 안 넣는다**: 5개에 동일 푸터를 박으면 같은 이유로 보일러플레이트. 각 템플릿이 목적에 맞는 마무리 섹션(링크 버튼/위치 문단 등)으로 끝낸다. 브랜딩 워터마크가 필요하면 그건 템플릿이 아니라 공개 레이아웃 차원의 별개 작업.
- **왼쪽 패널은 진입 시 기본 펼침 + 화살표 토글**(항상 접근 가능, 작업 중에도). 적용은 **기존 섹션 덮어쓰기 + 실행취소 토스트**(삭제와 같은 act-first 패턴, 확인 모달 없음). 주 사용처가 빈 페이지라 덮어쓸 게 없는 경우가 대부분이고, 있더라도 토스트로 복원.

### 인라인 편집 (노션 방식)

- 평소: 결과물 그대로 보임 (편집 UI 숨김)
- 섹션 클릭 → 편집 UI 활성화 (텍스트 입력 + 스타일 패널)
- 외부 클릭 → 편집 UI 닫힘
- 섹션 사이 호버 → `[+ 섹션 추가]` 버튼 나타남
- 미리보기 토글 별도 없음 (평소 화면 = 미리보기)
- 텍스트 콘텐츠는 인라인 편집, 버튼처럼 컨트롤 성격인 값(라벨·URL)은 스타일 패널에서 입력

### 스타일 패널

- 섹션 타입별로 컨트롤 분기(텍스트/구분선/버튼). 공용 색상·정렬 필드는 별도 컴포넌트로 공유.
- **콘텐츠/배경 탭**으로 분리. 콘텐츠 탭은 타입별 스타일, 배경 탭은 모든 섹션 공통 컨테이너(배경색·배경이미지). 색상 필드는 단색·그레디언트를 한 곳에서 고름(`utils/colorFill.ts`로 적용 분기).
- 우측 **도킹 드로어**(폭 `w-80`, 폭 애니메이션)로 띄움 → 섹션 선택 시 작업영역이 패널만큼 줄어듦. 위에 떠 가리는 오버레이가 아니라 자리를 차지하는 방식이라, 콘텐츠 폭이 넓어져도 패널과 겹치지 않음(처음엔 오버레이였으나 겹침 위험으로 도킹으로 전환).
- 텍스트 섹션은 표시(`<h1>`~`<p>`)와 편집(`textarea`)을 보이지 않는 미러로 같은 박스 높이에 맞춰, 선택/해제 시 레이아웃 시프트가 없게 함.

### 섹션 조작

- 이동: 드래그(dnd-kit). 큰 섹션이 이웃을 덮어 지저분해지지 않게 `DragOverlay`로 떠 있는 복제본을 띄우고 원본 자리는 비워 이웃이 부드럽게 자리를 내준다. 패널 항목 정렬도 같은 트랜지션 공유
- 추가: 추가 즉시 우측 드로어가 자동으로 열리고(그 섹션 선택), 텍스트는 바로 편집 포커스가 간다. 빈 이미지/갤러리/카드는 첫 업로드가 캔버스 드롭존이라 드로어를 열지 않는다.
- 삭제: 🗑 버튼(호버 또는 선택 시 노출). 누르면 **즉시 삭제 + 실행취소 토스트**. 확인 모달이 아니라 act-first — 가만히 두면 확정, 토스트의 "실행취소"로 같은 자리에 복원. 실행취소가 없는 모달의 마찰을 없애고 안전망은 토스트가 맡는다. 콘텐츠 유무·타입 무관하게 **항상** 같은 흐름(예외 없음 = 예측 가능). 이미지 정리(`deleteImage`)는 되돌릴 수 없게 토스트가 닫힐 때만 실행해 실행취소 시 이미지가 보존된다
- 복제: 아직 안 만듦

### 자동 저장 정책

- **텍스트/스타일 변경**: 3초 디바운스 자동저장, 변경 있을 때만(`isDirty` 플래그)
- **이미지 첨부**: 파일 선택 즉시 자동 업로드(메모리 미리보기 + 백그라운드 Cloudinary 업로드). 별도 "저장" 버튼 없음
- 업로드 성공 시 sections에 URL 반영 → **텍스트/스타일과 동일하게** 자동저장(3초)으로 DB 반영. 저장 상태 표시도 동일(즉시 flush 안 함 → "저장되지 않은 변경사항"이 보였다가 저장됨)
- 스토리지 관리는 "명시적 저장"이 아니라 **자동 삭제**로: 교체 시 옛 이미지 삭제, 섹션 삭제 시 해당 이미지 삭제(`deleteImage`). 안 쓴 이미지가 실시간 정리됨. 자동저장 전 이탈 시의 잔여 orphan은 이탈 flush + 추후 cron이 백스톱
- 이유: "올리고 또 버튼 누르기"는 UX 군더더기. 자동 업로드+자동 삭제가 버튼 없이도 용량을 깨끗하게 유지

### 상태 표시

- 디바운스 자동저장 직후: "자동 저장됨"
- 저장 버튼 클릭 직후: "저장됨" (자동저장과 구분 — `markSaved(snapshot, manual)`, `SaveStatus`에 `manualSaved`)
- 저장 안 됨 (디바운스 진행중 / 이미지 업로드 중): "저장되지 않은 변경사항 있음" + "저장" 버튼 강조
- 처음 상태와 같으면 표시 없음(`idle`)

### 페이지 이탈 경고

`isDirty = true` 상태로 나가려 할 때:

- 앱 내 이탈(헤더 나가기 버튼 + 브라우저 뒤로가기) → **shadcn(base-ui) 확인 모달**. 자동저장 모델이라 '버리기' 개념이 없어 "저장하고 나가기"로 명시(저장 후 이동). 브라우저 뒤로가기는 `popstate`로 가로채 모달을 띄움(센티넬 history 사용).
- 탭 닫기/새로고침 → `beforeunload` 네이티브 경고. 브라우저 보안 정책상 언로드 시점엔 커스텀 UI를 못 띄워 네이티브만 가능(텍스트·스타일 변경 불가).
- 안전망: 그 외 경로로 에디터가 언마운트돼도 `saveNow()`로 미저장분을 백그라운드 저장.

### 다크모드 (에디터 크롬 vs 캔버스)

에디터는 두 레이어를 분리한다.

- **크롬 (헤더 + 스타일 패널 등 작업 도구 UI)**: 앱 테마(라이트/다크)를 따른다. 헤더·대시보드와 동일하게 토큰 기반으로 자동 반영.
- **캔버스 (작업 공간)**: 앱 다크모드와 **절연**한다. 캔버스는 사용자가 만드는 실제 페이지라 배경색·이미지를 직접 정하고, 공개 페이지는 자체 다크모드 토글을 따로 가진다. 따라서 에디터를 다크로 보더라도 캔버스는 그 페이지가 실제로 보일 모습 그대로 렌더한다.

구현됨(Phase 5):

- `.canvas-light` 클래스가 `:root` 라이트 토큰을 다시 적용해 `.dark` 아래여도 라이트로 절연한다. `color-scheme: light`로 폼 컨트롤, 스크롤바 등 시스템 색도 고정. 캔버스, 에디터 미리보기, 공개 페이지(`(public)/layout.tsx`)에 적용.
- 텍스트 기본색(빈 문자열)은 페이지 기본 글자색(라이트 절연된 `--foreground` = 검정)으로 의미를 둔다. 스타일 패널 색상 스와치도 채움색만 `canvas-light`로 렌더해 패널이 다크여도 기본색이 안 뒤집힌다.
- 공개 페이지는 지금은 항상 라이트. 사용자가 고르는 공개 페이지 자체 다크 토글은 아직 안 만듦.

### 다크모드 토글 배치

다크모드는 Blanoir의 작업 도구(크롬)에만 영향을 주고 결과물(공개 페이지·캔버스·미리보기)은 항상 라이트로 절연되므로, 토글은 "색이 실제로 바뀌는 화면"에만 둔다. 토글은 한 번 정하면 next-themes가 localStorage에 저장해 유지되는 "한 번 설정" 값이다.

- **Blanoir UI 화면(랜딩·로그인·회원가입·대시보드·에디터)**: 헤더에 토글. 한 화면에 하나씩 항상 같은 자리라 중복이 아니라 일관성이다.
- **결과물·미리보기(공개 페이지·캔버스)**: 토글 없음. 어차피 라이트 고정이라 의미가 없다.
- **로그인 사용자**: 위 헤더 토글과 별개로 설정 페이지(`/settings`)에서도 테마를 고를 수 있다. 테마는 개인 취향 설정이라 계정 설정 안에 한 항목으로 같이 둔다.
- 대시보드 헤더는 로그아웃 버튼을 빼고 설정 아이콘으로 대체했고, 로그아웃은 설정 페이지 안으로 옮겼다. 토글은 그대로 유지.

## MVP 섹션 8개

1. **제목** (title) — 글자 크기 4단계로 `h1`~`p` 자동 매핑 (SEO 위계)
2. **문단** (paragraph) — 긴 글
3. **이미지** (image) — 1장
4. **버튼** (button) — 외부 링크
5. **구분선** (divider)
6. **공백** (spacer) — 섹션 사이 여백 (높이 3단계)
7. **갤러리** (gallery) — 여러 이미지 그리드
8. **카드** (card) — 이미지+제목+설명 묶음 (메뉴/프로젝트 소개용)

## 이미지 업로드 정책

- 업로드 시점: **파일 선택 즉시** 백그라운드 Cloudinary 업로드(스피너). 성공하면 sections에 URL 반영 → 자동저장(3초)으로 DB 반영. 공용 훅 `hooks/useImageUpload.ts`(`uploadOne`/`uploadMany`)로 검증·업로드·진행 플래그를 일원화
- **조작 위치 일원화**: 빈 상태의 **첫 업로드만 캔버스 드롭존**, 이미지가 생긴 뒤 **변경·교체·삭제·추가는 스타일 패널에서**(캔버스는 결과물 표시 전용). 이미지 섹션은 패널 썸네일+업로드/제거 아이콘, 갤러리는 패널 목록(드래그 순서 + 행별 교체/제거) + "이미지 추가" 버튼
- 업로드 중에는 스타일 드로어를 잠가(`pointer-events-none`) 미완료 상태 변경을 막음
- `alt`은 업로드 파일명에서 자동 생성(수동 입력칸 없음)
- 안 쓴 이미지 정리: 교체·삭제·섹션 삭제·페이지 삭제 시 `deleteImage`로 **즉시 삭제**(본인 `blanoir/{userId}` 폴더만). 잔여 orphan은 추후 cron이 백스톱
- 파일 크기 제한: 파일당 **5MB**

## SEO (MVP는 자동 생성만)

| 메타데이터         | 자동 생성 규칙                      |
| ------------------ | ----------------------------------- |
| `<title>`          | 페이지 제목 (사용자 입력)           |
| `meta description` | 첫 번째 문단 텍스트 앞 150자        |
| `og:image`         | 첫 번째 이미지 (없으면 기본 이미지) |

- Server Components에서 `generateMetadata` 활용
- 사용자가 SEO 직접 입력 → 아직 안 만듦
- 사이트 전역 메타(`title` 템플릿·`openGraph`·`twitter`·`robots`)는 루트 `layout`에서, 사이트 상수는 `lib/site.ts`로 모음. 베이스 URL은 `NEXT_PUBLIC_SITE_URL`(미설정 시 localhost 폴백)
- `robots.txt`(`app/robots.ts`)는 결과물·랜딩만 색인 허용, 대시보드·에디터·설정·인증·`api`는 disallow. 인증/에디터 라우트는 `noindex`
- `sitemap.xml`(`app/sitemap.ts`)은 **공개 페이지를 DB에서 동적 수집**(`force-dynamic`) — 랜딩 + 공개된 `/user/[handle]/[pageId]`. **공개 사용자 페이지는 검색 색인 대상**(결과물이 검색에 노출되는 게 서비스 가치)
- 검색엔진 소유권 인증은 메타태그 방식(`GOOGLE_SITE_VERIFICATION`·`NAVER_SITE_VERIFICATION` env → 루트 `layout` verification). 구글 서치콘솔·네이버 서치어드바이저에 sitemap 제출

## 배포 / 호스팅

- **Vercel + `blanoir.vercel.app`**. 도메인은 데모라 **구매하지 않고** 기본 vercel.app 주소로 운영. 추후 도메인 구매 시 `NEXT_PUBLIC_SITE_URL`·OAuth Redirect·검색엔진 인증만 교체하면 됨
- `NEXTAUTH_URL`은 Auth.js v5가 Vercel에서 자동 추론하지만, 배포 도메인이 여러 개(프로덕션·`git-main`·해시)라 콜백이 항상 정식 주소와 맞도록 `https://blanoir.vercel.app`로 **명시**
- **함수 리전을 서울(`icn1`)로 고정** — Atlas 클러스터(`ap-northeast-2`)·한국 사용자와 같은 리전이라야 매 요청 DB 왕복 지연이 작다. 기본값(미국 `iad1`)이면 태평양 왕복이 누적돼 체감 속도가 느림
- 이미지 업로드 Server Action은 기본 본문 제한 1MB라 5MB 이미지에서 500이 났다 → `next.config.ts`의 `serverActions.bodySizeLimit`을 6MB로 상향(5MB 한도 + multipart 오버헤드)
- Atlas Network Access는 Vercel IP가 비고정이라 `0.0.0.0/0` 허용. 실제 방어는 DB 유저 credential(`MONGODB_URI`)이 담당

## 페이지 공개/비공개

- 기본값: **비공개** (작업 중)
- 사용자가 "공개" 토글 → 누구나 접근 가능
- 비공개 페이지: **본인 포함 누구나 404** (공개 URL 자체가 죽음 — "비공개인데 왜 보이지" 혼란 제거, 멘탈 모델 단순). "공개 전 결과물 확인"은 비공개 URL을 여는 대신 **에디터 안 미리보기**(헤더/공개 드로어, 실제 `PublicSection` 렌더)로 제공
- 프로필 `/user/[handle]` 페이지 목록: **공개 페이지만 표시**(본인이 봐도 공개만). 본인의 비공개 페이지는 **대시보드**에서 관리·편집 (프로필은 공개 쇼케이스, 대시보드는 소유자 관리 공간으로 역할 분리)

## 설정 페이지

`/settings`는 로그인 사용자의 계정 관리 공간. 진입은 대시보드 헤더의 설정 아이콘. 다음을 한곳에서 다룬다.

- **프로필**: 닉네임, 페이지 주소(handle), 프로필 이미지. handle을 바꾸면 그 사용자의 모든 공개 페이지 주소가 함께 바뀐다(옛 주소 링크는 끊김 — 데모라 리다이렉트 없이 감수). 이미지는 Cloudinary 업로드, 없으면 기본 인물 아이콘. 조작은 이미지 표시 전용 + 버튼으로 일원화(에디터 패턴과 동일).
- **계정**: 이메일·로그인 방식(소셜/이메일)·가입일 표시(읽기 전용).
- **비밀번호**: 로컬 계정만 현재 비밀번호 확인 후 변경. 소셜 계정은 안내 문구.
- **화면**: 라이트/다크 테마 선택.
- **요금제**: Free/Pro 카드. 결제는 아직 미구현이라 누르면 "준비 중" 토스트만 띄운다. Blanoir의 무료 정체성은 유지하되 향후 고급 기능용 유료 플랜 자리만 잡아둔 것.
- **로그아웃**: 페이지 하단.

닉네임/handle/비번 변경은 설정 페이지에서 구현됨. 이메일 인증은 아직 안 만듦.

## 에러 / 예외 처리

| 케이스                         | 처리                                                            |
| ------------------------------ | --------------------------------------------------------------- |
| 시스템 에러 (렌더링)           | `error.tsx` (Error Boundary)                                    |
| 404 (페이지 없음)              | `not-found.tsx`, `notFound()` 함수                              |
| 비공개 페이지 접근 (본인 포함) | 404 (보안 + 멘탈 모델 단순)                                     |
| 없는 사용자 URL                | 404                                                             |
| 비로그인 에디터 접근           | `proxy.ts`(미들웨어)에서 로그인 페이지 리다이렉트               |
| 조회/변경 실패                 | 서버 액션이 `{ ok, message }` 반환 → 호출부에서 토스트 (Sonner) |
| 저장 실패                      | 동일 (토스트)                                                   |

### 코드 구조

조회는 서버 컴포넌트에서 직접 DB 호출(SSR), 변경은 서버 액션 직접 호출. 별도 클라이언트 데이터 패칭 레이어(React Query 등)는 두지 않는다.

```ts
// 서버 액션은 예외를 try/catch로 잡아 결과 객체로 반환
export async function savePage(...): Promise<{ ok: true } | { ok: false; message: string }> {
  try {
    // ...작업
    return { ok: true }
  } catch (error) {
    console.error(error)
    return { ok: false, message: '잠시 후 다시 시도해 주세요' }
  }
}
```

호출부(폼/버튼)에서 `if (!result.ok) toast.error(result.message)`로 안내한다. 렌더링 중 발생하는 시스템 에러는 `error.tsx`(Error Boundary), 없는 경로는 `not-found.tsx`가 잡는다.
