# Blanoir 설계 결정 (DECISIONS)

> 무엇을 왜 만들기로 했나. 코드 작성 규칙은 `AGENT.md`, 현황/작업순서는 `../CLAUDE.md` 참고.

## 프로젝트 개요

비개발자도 5분 안에 자기 홈페이지를 만들고 배포할 수 있는 **한글 노코드 페이지 빌더**.

## 서비스 정체성

### 서비스명: **Blanoir** (블라누아)

**의미:**

- 프랑스어 `Blanc`(흰색) + `Noir`(검정)의 합성어
- "흰 캔버스 위에 검정 잉크로 표현하다"
- "빈 페이지에 당신만의 이야기를 새기다"
- 모노톤 디자인 철학 직접 담음

**한 줄 소개:**

> "5분 만에 만드는 내 홈페이지. 코딩 모르는 사람도, 한글 클릭만으로, 무료로."

### 타겟 사용자

타겟 안 좁힘 = "코딩 모르는데 자기 페이지 만들고 싶은 모든 사람"

- 자기소개/포트폴리오
- 카페/매장 소개
- 결혼식 청첩장
- 학생 이력서
- 취미/개인 페이지

### 차별점

| 비교 대상     | 우리 차별점                |
| ------------- | -------------------------- |
| 아임웹/식스샵 | 가볍고 무료, 비개발자 친화 |
| 노션 사이트   | 디자인 자유로움, 한글 특화 |
| Wix/Webflow   | 가벼움, 한글, 무료         |

핵심 가치: **"학습 없이 직관적으로"** + **무료** + **한글** + **빠른 배포**

## 기술 선택 이유

> 스택 목록은 `../CLAUDE.md` 참고. 여기는 왜 그렇게 골랐는지.

### 풀스택 선택 이유

- 우리 서비스 백엔드 가벼움 (CRUD + 인증)
- 개발 속도, 단순함, 비용 0원
- SSR로 페이지 조회 처리 자연스러움 (DB 직접 호출)

### 서버리스 커넥션 관리 (중요)

Vercel은 서버리스 — 함수가 요청마다 켜졌다 꺼짐. DB 커넥션 캐싱이 필수.

- 캐싱 없으면: 호출마다 `mongoose.connect()` → 커넥션 누수 → MongoDB Atlas 무료(M0) 한도(약 500개) 초과 시 연결 실패
- 해결: `lib/mongodb.ts`에서 `global` 객체에 커넥션 캐싱 후 재사용 (Next.js + Mongoose 공식 표준 패턴)
- 옵션 아니라 정석 — 트래픽 적어도 무조건 적용
- Blanoir는 단순 CRUD라 무거운 집계·크롤러가 없어 커넥션 폭증 위험 자체가 낮음. 캐싱 패턴만 지키면 충분

### 비용 구성

| 영역      | 서비스             | 비용       |

| 프론트+백 | Vercel Hobby       | 0원        |

## 데이터 모델 (MongoDB)

### User 스키마

```ts
{
  _id: ObjectId,
  email: String,           // 로그인 이메일 (unique)
  password: String,        // 로컬 가입시만 (bcrypt 해시). 소셜 가입은 null
  resetCode: String,       // 비번 재설정 인증코드 (bcrypt 해시). 미발급시 null
  resetCodeExpiresAt: Date,// 인증코드 만료 시각
  resetCodeAttempts: Number,// 인증코드 검증 실패 횟수 (5회 초과 시 코드 폐기)
  name: String,            // 닉네임 (한글 OK, 화면 표시용)
  handle: String,          // URL용 (영문+숫자+언더바, 자동 생성)
  profileImage: String,    // 프로필 이미지 URL (소셜은 자동, 로컬은 기본 이미지)
  provider: String,        // "google" | "kakao" | "naver" | "local"
  createdAt: Date,
  updatedAt: Date
}
```

### Page 스키마

```ts
{
  _id: ObjectId,
  pageId: String,          // nanoid 10자 (URL용)
  userId: ObjectId,        // 작성자
  title: String,           // 페이지 제목 (한글 OK, 자유롭게)
  sections: Section[],     // 섹션 데이터 배열
  isPublic: Boolean,       // 공개/비공개 (기본 false)
  createdAt: Date,
  updatedAt: Date
}
```

### handle 자동 생성

닉네임 입력 시 → 영문 로마자 변환 → 중복시 숫자 붙임

| 닉네임        | handle        | URL                 |
| ------------- | ------------- | ------------------- |
| 김철수        | kimchulsoo    | /user/kimchulsoo    |
| 영희          | younghee      | /user/younghee      |
| Chulsoo Kim   | chulsoo_kim   | /user/chulsoo_kim   |
| 카페별빛      | cafe_byeolbit | /user/cafe_byeolbit |
| 김철수 (중복) | kimchulsoo_2  | /user/kimchulsoo_2  |

- 규칙: 영문 소문자 + 숫자 + 언더바
- V2: 사용자가 handle 직접 변경 가능

### pageId (페이지 식별자)

- **nanoid 10자** (예: `V1StGXR8_Z`)
- 추측 불가 (보안)
- 충돌 확률 거의 0
- 노션과 유사한 패턴

## 회원가입 / 로그인 흐름

### 소셜 로그인 (구글/카카오/네이버) — 원클릭 가입

1. "구글로 로그인" (또는 카카오/네이버) 클릭
2. 소셜 인증 통과
3. 자동 가입:
   - name = 소셜 이름 그대로
   - handle = name 기반 자동 변환 + 중복시 숫자
   - profileImage = 소셜 사진 그대로
   - email = 소셜 이메일
   - provider = "google" | "kakao" | "naver"
4. 바로 대시보드로 이동

### 로컬 회원가입 (이메일/비번) — 폼 입력

1. 회원가입 페이지 진입
2. 입력: 이메일 + 비번 + 닉네임 (3개)
3. 검증:
   - 이메일 형식 + 중복 체크
   - 비번 8자 이상
   - 닉네임 한글 OK
4. 비번 bcrypt 해싱 → DB 저장
5. handle 자동 생성
6. provider = "local"
7. 자동 로그인 → 대시보드

### 로컬 로그인

- NextAuth `CredentialsProvider` 사용
- 이메일 + 비번 → bcrypt 검증 → JWT 발급

### 비밀번호 재설정 (MVP 포함) — 이메일 인증코드 방식

1. 사용자가 이메일 입력
2. 이메일이 DB에 있으면(local 계정) → **6자리 숫자 인증코드 생성**
3. 인증코드 bcrypt 해싱 → DB의 `resetCode` 저장 + `resetCodeExpiresAt`(10분) 설정
4. 인증코드를 이메일로 발송 (Gmail SMTP / nodemailer, 키 없으면 콘솔 폴백)
5. 사용자가 인증코드 입력 → 검증 (일치 + 미만료)
6. 통과 시 새 비밀번호 입력 → bcrypt 해싱 → `password` 업데이트 + `resetCode`/`resetCodeExpiresAt` 폐기

> 미가입/소셜 계정이면 토스트로 안내하고 발송 중단 (데모 — UX 우선, 계정 열거 노출은 감수. 운영 시 항상 성공 응답으로 가릴 수 있음)
> 이메일 = ID이므로 **아이디 찾기는 없음**
> 인증코드 **5회 오입력 시 폐기**(brute-force 방지) — `resetCodeAttempts`로 카운트
> 이메일 인증(가입 시)은 V2, 닉네임/handle 변경은 V2 설정 페이지에서

## 에디터 UX

### 입력 방식: 카드/섹션 추가 (아임웹 스타일이지만 더 가볍게)

- `[+ 새 섹션 추가]` 버튼 → 종류 메뉴 (제목/문단/이미지/버튼/구분선/갤러리/카드)
- 섹션 추가시 캔버스에 표시
- 학습 0, 클릭만으로 작동

### 인라인 편집 (노션 방식)

- 평소: 결과물 그대로 보임 (편집 UI 숨김)
- 섹션 클릭 → 편집 UI 활성화 (텍스트 입력 + 스타일 패널)
- 외부 클릭 → 편집 UI 닫힘
- 섹션 사이 호버 → `[+ 섹션 추가]` 버튼 나타남
- 미리보기 토글 별도 없음 (평소 화면 = 미리보기)

### 섹션 조작

- 이동: 위/아래 화살표 또는 드래그(dnd-kit)
- 삭제: 🗑 버튼
- 복제: V2

### 자동 저장 정책

- **텍스트/스타일 변경**: 5초 디바운스 자동저장, 변경 있을 때만(`isDirty` 플래그)
- **이미지 첨부**: 프론트 미리보기만(메모리), **사용자 "저장" 버튼 클릭시 Cloudinary 업로드**
- 이유: 자동 업로드시 안 쓴 이미지가 스토리지 차지 → 명시적 저장으로 용량 관리

### 상태 표시

- 평소: "자동 저장됨"
- 저장 안 됨 (디바운스 진행중 / 이미지 미업로드): "저장되지 않은 변경사항 있음" + "저장" 버튼 강조

### 페이지 이탈 경고

`isDirty = true` 상태로 나가려 할 때:

- 앱 내 이탈(헤더 나가기 버튼 + 브라우저 뒤로가기) → **shadcn(base-ui) 확인 모달**. 자동저장 모델이라 '버리기' 개념이 없어 "저장하고 나가기"로 명시(저장 후 이동). 브라우저 뒤로가기는 `popstate`로 가로채 모달을 띄움(센티넬 history 사용).
- 탭 닫기/새로고침 → `beforeunload` 네이티브 경고. 브라우저 보안 정책상 언로드 시점엔 커스텀 UI를 못 띄워 네이티브만 가능(텍스트·스타일 변경 불가).
- 안전망: 그 외 경로로 에디터가 언마운트돼도 `saveNow()`로 미저장분을 백그라운드 저장.

### 코드 보기

- **MVP에 안 넣음** (V2)
- 비개발자 친화 본질에 어긋남
- 추후 V2에서 개발자용 모드로 추가 가능

### 미리보기

- PC/모바일 토글 (MVP 포함)
- 모바일 스타일은 **자동 반응형** (Tailwind 반응형 클래스)
- 사용자가 모바일 전용 스타일 따로 입력 X → V2

### 다크모드 (에디터 크롬 vs 캔버스)

에디터는 두 레이어를 분리한다.

- **크롬 (헤더 + 스타일 패널 등 작업 도구 UI)**: 앱 테마(라이트/다크)를 따른다. 헤더·대시보드와 동일하게 토큰 기반으로 자동 반영.
- **캔버스 (작업 공간)**: 앱 다크모드와 **절연**한다. 캔버스는 사용자가 만드는 실제 페이지라 배경색·이미지를 직접 정하고, 공개 페이지는 자체 다크모드 토글을 따로 가진다. 따라서 에디터를 다크로 보더라도 캔버스는 그 페이지가 실제로 보일 모습 그대로 렌더한다.

구현은 Phase 5의 다크 토글과 함께 진행:

- 캔버스 컨테이너에 라이트 토큰을 고정하거나 페이지 자체 배경/전경색으로 렌더해 `.dark` 상속을 끊는다.
- 텍스트 기본색(빈 문자열)은 에디터 테마색(`--foreground`)이 아니라 페이지 기본 글자색으로 의미를 둔다.

## 이미지 업로드 정책

- 첨부 시점: 프론트 메모리에만 보관 (base64/ObjectURL 미리보기)
- 업로드 시점: 사용자 "저장" 버튼 클릭시 Cloudinary 업로드
- 업로드 후: sections에 URL 박고 MongoDB 저장
- 파일 크기 제한: 파일당 **5MB**
- 안 쓴 이미지 정리: V2 cron으로 자동 삭제

## SEO (MVP는 자동 생성만)

| 메타데이터         | 자동 생성 규칙                      |
| ------------------ | ----------------------------------- |
| `<title>`          | 페이지 제목 (사용자 입력)           |
| `meta description` | 첫 번째 문단 텍스트 앞 150자        |
| `og:image`         | 첫 번째 이미지 (없으면 기본 이미지) |

- Server Components에서 `generateMetadata` 활용
- 사용자가 SEO 직접 입력 → V2

## 페이지 공개/비공개

- 기본값: **비공개** (작업 중)
- 사용자가 "공개" 토글 → 누구나 접근 가능
- 비공개 페이지:
  - 본인: 정상 접근
  - 다른 사람: **404 반환** (보안 위해 "비공개임"도 안 알려줌)
- 본인 프로필 `/user/[handle]`에 페이지 목록 표시 시:
  - 본인 보기: 공개+비공개 다 표시
  - 다른 사람: 공개만

## 에러 / 예외 처리

| 케이스                       | 처리                                               |
| ---------------------------- | -------------------------------------------------- |
| 시스템 에러 (렌더링)         | `error.tsx` (Error Boundary)                       |
| 404 (페이지 없음)            | `not-found.tsx`, `notFound()` 함수                 |
| 비공개 페이지 다른 사람 접근 | 404 (보안)                                         |
| 없는 사용자 URL              | 404                                                |
| 비로그인 에디터 접근         | `middleware.ts`에서 로그인 페이지 리다이렉트       |
| 조회/변경 실패               | React Query `onError` + Toast 알림 (shadcn Sonner) |
| 저장 실패                    | React Query `onError` + 토스트                     |

### 코드 구조

```ts
// React Query 전역 설정
const queryClient = new QueryClient({
  defaultOptions: {
    queries: { retry: 1, staleTime: 60_000 },
    mutations: { onError: () => toast.error('저장 실패') },
  },
})
```

Server Action 내부 에러는 try/catch로 잡고 throw → React Query `onError`에서 토스트 표시.
